# YouBike Demand Forecasting PRD

## 問題陳述（Problem Statement）
台北市 YouBike 2.0 擁有超過千個站點，尖峰時段常出現缺車或滿車的痛點。營運團隊缺乏高解析度的短期需求預測工具，導致調度排程仰賴經驗法則，效率不彰。本專案旨在建立一套以資料為核心的預測服務，縮短決策時間並降低調度成本。

## 使用者故事（User Stories）
1. **營運調度員**：希望在班前即可查看各站點於未來 15、30、60 分鐘的可借數量趨勢，便於規劃補車與回收動線。
2. **產品經理**：需要即時掌握模型預測誤差與風險情境，才能與市府與合作夥伴協調資源。
3. **資料科學家**：需要明確的資料規格、特徵定義與驗收標準，以便迭代模型與比較不同演算法。
4. **視覺化工程師**：需取得站點圖層、時間滑桿與模型輸出格式，打造 deck.gl 互動展示。

## KPI 與量化指標
- 尖峰時段 T+15 MAE ≤ 2.0。
- 離峰時段 T+30 MAE ≤ 2.5。
- 全日 T+60 MAE ≤ 3.0。
- 缺車／滿車站點比例下降 5%。
- 每週生成預測報告並透過 CI 上傳 `reports/metrics.json`。

## 成功定義（Definition of Done）
- ETL 可以穩定蒐集每分鐘資料，並建立最新站點圖。
- LSTM baseline 在歷史資料上可重現驗收門檻 ±10% 的精度。
- deck.gl demo 可切換不同預測視窗並顯示站點資訊。
- CI 會自動安裝依賴、驗證規格檔案存在、執行 baseline 評估與單元測試。

## 預估里程碑（Milestones）
1. **週 1-2：ETL 與資料探索** — 完成資料抓取、清洗、初步分析與可視化。
2. **週 3：Baseline 模型** — 建立 LSTM baseline、產出初版評估報告。
3. **週 4：deck.gl Demo** — 與設計師合作完成時間滑桿、殘差圖層與互動介面。
4. **週 5-6：圖神經網路模型** — 實作 DCRNN 與 Graph WaveNet，與 baseline 比較。
5. **週 7：驗收與上線準備** — 彙整評估、風險與備援方案，準備上線文檔。

## 風險與假設
- 假設官方 API 可穩定存取，若出現頻繁斷線需尋找備援資料來源。
- 若無法取得完整站點歷史，需與營運端協調補資料或使用相似站點推估。
- 模型訓練依賴 GPU，需預先規劃雲端運算資源或 On-prem 硬體。

## 不在範圍（Out of Scope）
- 長期 (>7 天) 需求預測。
- 與金流、會員系統的整合需求。
- 自動化調度派車系統，僅提供決策支援資訊。
